# 2.5.1. Diagrama de Sequências (Materiais)

Como iniciativa extra, o Marcus e Júlio fizeram o diagrama de sequência exemplificando como funcionará a lógica de operações com materiais na aplicação (documentos, imagens, etc.). O diagrama de sequências representa as linhas de vida de partes do sistema e suas mensagens trocadas, é uma linha do tempo (da esquerda para a direita, de cima para baixo), o diagrama também apresenta numerações para identificar a ordem de algumas operações. O diagrama foi feito usando [PlantUML](https://plantuml.com/) no VSCode.


![Diagrama de Sequências](./img/sequencia-materiais.png)

<p align="center"><b>Fonte:</b> Autoria de <a href="https://github.com/julnox">Júlio César</a> e <a href="https://github.com/MarcusVcd">Marcus Vinicius Cunha</a>.</p>


## Atores e Participantes

No diagrama está identificado um ator e 4 participantes, sendo eles:
- Usuário: o público alvo do nosso aplicativo, interagem com a View de forma visual (clicando, preenchendo);
- View: representa a primeira camada de interação, responsável por definir a estrutura e como será apresentado a tela ao usuário;
- Controller: representa a camada intermediária de interação entre a view e os services, responsável por identificar a view correta para determinado fluxo e o que será apresentado ao usuário;
- Service: responsável pela interação entre o controller e o banco de dados, realiza consultas ao banco e retorna dados tratados;
- Database: o banco de dados, onde os dados estão armazenados.

## Descrição

O diagrama representa todos os fluxos de operações com Materiais da aplicação:

- *Buscar Material*: o usuário clica em um material para visualizar, a view chama o método buscarMaterial informando o id do material (podendo estar presente na view como um input hidden), o controller chama o método obterMaterial(id) e o service manda uma consulta findById(id) pro banco de dados, a consulta deve ser rápida já que o id é único e deve estar indexado. A partir disto é retornado o material, o controller irá decidir o que a view terá acesso e qual é a view correta para renderizar, caso o banco de dados retorne null (não encontrou), será mostrado um modal de erro ao usuário;
- *Criar Material*: o usuário preenche os dados para criar (fazer upload) de um material e clica em "criar", a view chamará o método criarMaterial informando os dados preenchidos, o Controller deve estar responsável por tratar estes dados, mas é preciso ter um cuidado em todas as camadas para que esta funcionalidade não seja um ponto fraco para ataques XSS ou SQL Injection, o controller instancia um objeto (entidade ou DTO) para que o service insira no banco de dados com um comando de insert, o banco de dados irá retornar o id do novo material (identificando um sucesso ao inserir), neste caso será mostrado um modal de sucesso, se o banco retornar null (inserção falhou) será mostrado um modal de erro, se os detalhes do erro forem muito técnicos, será simplificado para o usuário, mas a aplicação terá um logger para avisar os desenvolvedores.
- *Excluir Material*: o usuário escolhe um material e clica em excluir, a view chama o método excluirMaterial informando o id do material, o controller chama o service e o service envia um comando de delete(id), o banco irá retornar o id do material excluído (informando um sucesso na exclusão) e será mostrado um modal de sucesso para o usuário, caso o banco retorne null será mostrado um modal de erro.
- *Baixar Material*: o usuário visualiza um material e clica no botão de baixar, a view chama o método baixarMaterial(id) do controller, o controller envia o método buscarArquivo(id) para o service que por sua vez envia uma consulta ao banco de dados para retornar o arquivo binário armazenado, o arquivo será enviado em binário para o controller que irá se encarregar de enviar o arquivo para download na view.
- *Editar Material*: o processo é o mesmo em que o criar material, com a exceção que o id do material é informado e o banco de dados perfoma um update().
