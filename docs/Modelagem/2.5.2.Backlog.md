# 2.5.2. Backlog do Produto

## O que é o backlog?

O backlog de produto é uma lista priorizada e dinâmica de requisitos, funcionalidades e melhorias que orienta o desenvolvimento ao longo do projeto. No contexto do OrganizeSeuGrupo, ele foi estruturado a partir dos artefatos produzidos nas fases iniciais, como o [Design Sprint](https://unbarqdsw2026-1-turma02.github.io/2026.1-T02-G2_OrganizeSeuGrupo/#/Base/1.1.DesignSprint) e o [BPMN](https://unbarqdsw2026-1-turma02.github.io/2026.1-T02-G2_OrganizeSeuGrupo/#/./Base/1.3.ModelagemBPMN), priorizando os fluxos de criação de grupos, agendamento de reuniões e integração com calendários externos como núcleo da plataforma.

Os itens do backlog são organizados em três níveis hierárquicos. O nível mais alto são os **Temas**, que agrupam grandes áreas do sistema. Dentro de cada tema existem os **Épicos**, que representam funcionalidades amplas. Por fim, as **User Stories (US)** descrevem necessidades específicas dos usuários no formato *"Eu, como [agente], quero [ação], para [benefício]"*. Esse formato é detalhado com mais profundidade durante o Sprint Planning.

## Requisitos funcionais:

Os requisitos funcionais foram levantados a partir das atividades do [Design Sprint](https://unbarqdsw2026-1-turma02.github.io/2026.1-T02-G2_OrganizeSeuGrupo/#/Base/1.1.DesignSprint) e dos fluxos mapeados no [BPMN](https://unbarqdsw2026-1-turma02.github.io/2026.1-T02-G2_OrganizeSeuGrupo/#/./Base/1.3.ModelagemBPMN). As funcionalidades identificadas nesses artefatos foram traduzidas nos requisitos apresentados na tabela a seguir, cobrindo desde o cadastro e autenticação de usuários até a integração com o Google Calendar.

<p style="text-align:center">Tabela 4: Requisitos funcionais</p>

| Código | Título |
| ------ | ------ |
| RF01 | O sistema deve permitir o cadastro de novos usuários mediante preenchimento de dados básicos e seleção de perfil acadêmico. |
| RF02 | O sistema deve autenticar usuários registrados por e-mail e senha, diferenciando os níveis de acesso entre Usuário Padrão e Administrador. |
| RF03 | O sistema deve permitir que o usuário visualize e edite suas informações de perfil, incluindo dados pessoais e acadêmicos. |
| RF04 | O sistema deve permitir a criação de Grupos de Estudo mediante preenchimento de campos obrigatórios (nome, disciplina, tags e descrição). |
| RF05 | O sistema deve disponibilizar mecanismo de busca e filtragem de grupos por etiquetas (tags) e palavras-chave. |
| RF06 | O sistema deve permitir que um usuário externo envie uma solicitação de entrada ao criador do grupo. |
| RF07 | O sistema deve permitir o agendamento de reuniões vinculadas a um grupo de estudo específico, armazenando os dados no banco. |
| RF08 | O sistema deve listar cronologicamente todas as reuniões associadas aos grupos nos quais o usuário está inserido. |
| RF09 | O sistema deve renderizar um componente de calendário interno compilando os horários e grupos do usuário autenticado. |
| RF10 | O sistema deve integrar-se à API do Google Calendar via OAuth 2.0 para exportar os dados de uma reunião diretamente para a agenda do usuário. |
| RF11 | O sistema deve possuir um módulo de notificações que alerte o usuário sobre novos convites, reuniões agendadas e atualizações de grupos. |
| RF12 | O sistema deve disponibilizar um painel administrativo para que o perfil Administrador gerencie grupos e emita comunicados globais. |

<p align="center"><b>Fonte: </b>Autoria de <a href="https://github.com/LucasAlves71">Lucas Alves</a> e <a href="https://github.com/Acciolyy">Thiago Viriato Accioly</a> </p>

## Requisitos não funcionais:

Os requisitos não funcionais foram organizados com base no modelo URPS+, que classifica essas características em categorias como Usabilidade, Confiabilidade, Desempenho, Suportabilidade e extensões para áreas específicas como Interoperabilidade. As categorias foram selecionadas de acordo com as preocupações técnicas centrais do OrganizeSeuGrupo, como a dependência da API do Google Calendar, a necessidade de uma interface acessível para diferentes perfis de usuário e a manutenibilidade do código ao longo do projeto.

### Usabilidade

<p style="text-align:center">Tabela 5: Requisitos não funcionais de Usabilidade</p>

| Código | Título |
| ------ | ------ |
| RQN01  | O fluxo de criação de reuniões e grupos deve seguir a sequência modelada nos diagramas BPMN do projeto, limitando o número de telas intermediárias a no máximo três etapas distintas. |
| RQN02  | As etiquetas (tags) de disciplinas devem possuir diferenciação visual por cores para agilizar a leitura durante a busca. |

<p align="center"><b>Fonte: </b>Autoria de <a href="https://github.com/LucasAlves71">Lucas Alves</a> e <a href="https://github.com/Acciolyy">Thiago Viriato Accioly</a> </p>

### Confiabilidade e Segurança

<p style="text-align:center">Tabela 6: Requisitos não funcionais de Confiabilidade e Segurança</p>

| Código | Título |
| ------ | ------ |
| RQN03  | A exportação para o Google Agenda deve usar o fluxo OAuth 2.0, garantindo que o sistema não armazene credenciais das contas Google dos usuários. |
| RQN04  | Ações destrutivas realizadas pelo Administrador (exclusão de grupos) devem exigir confirmação dupla e utilizar soft delete, preservando os dados por ao menos 30 dias antes da remoção definitiva. |

<p align="center"><b>Fonte: </b>Autoria de <a href="https://github.com/LucasAlves71">Lucas Alves</a> e <a href="https://github.com/Acciolyy">Thiago Viriato Accioly</a> </p>

### Desempenho

<p style="text-align:center">Tabela 7: Requisitos não funcionais de Desempenho</p>

| Código | Título |
| ------ | ------ |
| RQN05  | A sincronização com a API do Google Calendar deve ocorrer de forma assíncrona, sem bloquear a interface do usuário por mais de 2 segundos. |
| RQN06  | As consultas ao banco de dados para busca de grupos por etiquetas devem retornar resultados em até 600 milissegundos, medidos em ambiente de homologação com carga simulada de 50 usuários simultâneos. |

<p align="center"><b>Fonte: </b>Autoria de <a href="https://github.com/LucasAlves71">Lucas Alves</a> e <a href="https://github.com/Acciolyy">Thiago Viriato Accioly</a> </p>
